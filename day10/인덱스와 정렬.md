# 인덱스가 정렬에 미치는 영향

## 배경: 왜 인덱스가 정렬에 영향을 줄까?

인덱스가 정렬에 영향을 준다는 사실을 이해하려면, 먼저 **인덱스가 내부적으로 어떤 구조로 저장되는지**를 알아야 한다.

### B-Tree 구조와 정렬의 관계

대부분의 DB 인덱스는 **B-Tree(Balanced Tree)** 구조로 저장된다.

```
        [30]
       /    \
   [10,20]  [40,50]
   /  |  \   /  |  \
 [5][15][25][35][45][55]
```

B-Tree의 핵심 특징은 **노드 내 키(key)가 항상 정렬된 상태를 유지**한다는 것이다.  
즉, 인덱스를 생성하는 순간 해당 컬럼의 값들은 이미 **정렬된 상태**로 저장된다.

> 💡 이게 왜 중요한가?  
> DB가 `ORDER BY`를 처리할 때, 정렬되지 않은 데이터를 가져와서 직접 정렬하면 비용이 크다.  
> 하지만 인덱스를 활용하면 **이미 정렬된 순서대로 데이터를 읽을 수 있어** 별도 정렬 작업이 필요 없다.

---

## 인덱스가 정렬에 미치는 영향

### 1. `ORDER BY`에서의 최적화

인덱스가 걸린 컬럼으로 `ORDER BY`를 하면, DB는 인덱스를 순서대로 스캔하는 것만으로 정렬된 결과를 반환할 수 있다.

```sql
-- age 컬럼에 인덱스가 있는 경우
SELECT * FROM users ORDER BY age ASC;
```

- **인덱스 없음** → 전체 데이터를 읽은 후 → 메모리에서 정렬 (filesort 발생)
- **인덱스 있음** → 인덱스를 순서대로 스캔 → 이미 정렬된 결과 반환

`EXPLAIN`으로 확인하면:

| 상황 | Extra |
|------|-------|
| 인덱스 없음 | `Using filesort` |
| 인덱스 있음 | (filesort 없음) |

`Using filesort`는 정렬을 위한 추가 연산이 발생했다는 뜻으로, 인덱스를 활용하면 이를 피할 수 있다.

---

### 2. 범위 쿼리(`BETWEEN`, `>`, `<`)에서의 정렬

B-Tree는 값이 정렬된 상태이므로, 범위 조건도 효율적으로 처리된다.

```sql
SELECT * FROM orders WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'
ORDER BY created_at ASC;
```

B-Tree에서 시작점(`2024-01-01`)을 찾은 후, 끝점까지 **리프 노드를 순서대로 순회**하면 된다.  
이미 정렬된 구조이므로 `ORDER BY`도 추가 비용 없이 처리된다.

---

### 3. 복합 인덱스(Composite Index)와 정렬

복합 인덱스는 **컬럼 순서가 정렬 순서와 일치할 때** 가장 효율적이다.

```sql
-- 인덱스: (last_name, first_name)
SELECT * FROM users ORDER BY last_name ASC, first_name ASC;  -- ✅ 인덱스 활용
SELECT * FROM users ORDER BY first_name ASC, last_name ASC;  -- ❌ 인덱스 활용 불가
```

복합 인덱스는 `(last_name, first_name)` 순서로 정렬되어 저장된다.  
따라서 정렬 순서가 인덱스 컬럼 순서와 다르면 B-Tree를 순서대로 읽는 것이 불가능해져 결국 filesort가 발생한다.

---

### 4. 정렬 방향(ASC / DESC)

B-Tree는 양방향 순회가 가능하다. 즉, 인덱스를 앞에서 뒤로도, 뒤에서 앞으로도 읽을 수 있다.

```sql
SELECT * FROM users ORDER BY age ASC;   -- 인덱스 순방향 스캔
SELECT * FROM users ORDER BY age DESC;  -- 인덱스 역방향 스캔
```

두 쿼리 모두 인덱스를 활용할 수 있다.  
단, 복합 인덱스에서 컬럼마다 방향이 섞이면 활용이 어렵다.

```sql
-- 인덱스: (a ASC, b ASC)
ORDER BY a ASC, b ASC    -- ✅
ORDER BY a DESC, b DESC  -- ✅ (전체 역방향)
ORDER BY a ASC, b DESC   -- ❌ (방향이 섞임, filesort 발생)
```

---

## 정리

| 상황 | 인덱스 활용 여부 | 이유 |
|------|---------------|------|
| `ORDER BY` 인덱스 컬럼 | ✅ | B-Tree가 이미 정렬된 상태 |
| 범위 쿼리 + 같은 컬럼 `ORDER BY` | ✅ | 리프 노드 순회로 처리 가능 |
| 복합 인덱스와 컬럼 순서 일치 | ✅ | 인덱스 정렬 순서 그대로 활용 |
| 복합 인덱스와 컬럼 순서 불일치 | ❌ | B-Tree 순서와 요청 순서가 달라 filesort 필요 |
| 혼합 정렬 방향 (복합 인덱스) | ❌ | B-Tree 단방향 순회로는 처리 불가 |

> **핵심 원리**: 인덱스는 B-Tree로 저장되어 항상 정렬된 상태다.  
> DB가 정렬 결과를 요청받았을 때, 인덱스의 정렬 순서와 요청 순서가 일치하면 **별도 정렬 없이 인덱스를 순서대로 읽는 것만으로 결과를 반환**할 수 있다.
