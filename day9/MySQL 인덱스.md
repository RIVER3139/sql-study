## MySQL 인덱스

**클러스터 인덱스**
- 테이블 생성 시 자동 생성, PK 기준으로 데이터가 물리적으로 정렬되어 저장
- 데이터 저장 방식 자체 (테이블당 1개)

**보조 인덱스**
- 추가로 만드는 검색 목차
- 리프에 PK값만 저장 → PK로 클러스터 인덱스에서 실제 데이터 가져옴
- 실제 데이터 다 넣으면 색인이 터지니까 PK만 저장

**PK 없으면?**
- NOT NULL UNIQUE 컬럼 대체 → 그것도 없으면 숨겨진 rowid 자동생성 (접근 불가, 성능 나쁨)
- 실무에서는 항상 PK 만들기

## B+Tree

- B = Balanced, 데이터 쌓일 때 자동으로 균형 맞춤
- 위쪽 노드 = 길잡이만 (절반씩 날리며 탐색)
- 리프 노드 = 실제 데이터 (클러스터) 또는 PK값 (보조)
- 리프끼리 옆으로 연결 → 범위 검색 빠름
- INSERT할 때 B+Tree 규칙대로 자동 배치

## 인덱스 설계

**카디널리티** = 값이 얼마나 다양한가
- 높을수록 인덱스 효과 좋음 (이메일 ✅ 성별 ❌)

**복합 인덱스 순서**
- 1순위: 쿼리에서 어떤 컬럼 먼저 쓰냐
- 2순위: 카디널리티 높은 거 앞으로
