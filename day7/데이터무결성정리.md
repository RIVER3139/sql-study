# 📝 Day 7 추가 (04/05) — 데이터 무결성
---

## 🔥 핵심 한 줄 요약

> **데이터 무결성 = 데이터가 정확하고 일관된 상태를 유지하도록 보장하는 것**

---

## 1. 데이터 무결성이 왜 중요한가?

- 잘못된 데이터가 DB에 들어가면 서비스 전체에 영향
- 예: 나이에 -5가 들어가거나, 존재하지 않는 부서에 직원이 배정되거나
- **제약 조건**을 걸어서 잘못된 데이터가 아예 들어오지 못하게 막는 것!

---

## 2. 기본 제약 조건

| 제약 조건 | 역할 | 예시 |
|-----------|------|------|
| NOT NULL | 빈 값(NULL) 허용 안 함 | 이름은 반드시 있어야 함 |
| UNIQUE | 중복 값 허용 안 함 | 이메일은 고유해야 함 |
| PRIMARY KEY | NOT NULL + UNIQUE (테이블당 1개) | 고객ID |
| DEFAULT | 값 안 넣으면 기본값 자동 설정 | 가입일 기본값 = 오늘 |

```sql
CREATE TABLE 회원 (
    회원ID INT PRIMARY KEY,
    이름 VARCHAR(50) NOT NULL,
    이메일 VARCHAR(100) UNIQUE,
    가입일 DATE DEFAULT (CURRENT_DATE)
);
```

---

## 3. 외래 키 제약 조건 (FOREIGN KEY) ⭐

> **다른 테이블의 기본 키를 참조**하여 데이터 간의 관계를 보장

```sql
CREATE TABLE 주문 (
    주문ID INT PRIMARY KEY,
    고객ID INT,
    상품명 VARCHAR(100),
    FOREIGN KEY (고객ID) REFERENCES 고객(고객ID)
);
```

### 외래 키가 하는 일
- 고객 테이블에 **없는 고객ID**로 주문을 등록하려 하면 → **에러!**
- 주문이 있는 고객을 고객 테이블에서 삭제하려 하면 → **에러!**

### 삭제/수정 시 옵션

| 옵션 | 동작 |
|------|------|
| RESTRICT (기본값) | 참조되고 있으면 삭제/수정 불가 |
| CASCADE | 부모 삭제 시 자식도 함께 삭제 |
| SET NULL | 부모 삭제 시 자식의 외래 키를 NULL로 변경 |
| SET DEFAULT | 부모 삭제 시 자식의 외래 키를 기본값으로 변경 |

```sql
FOREIGN KEY (고객ID) REFERENCES 고객(고객ID)
    ON DELETE CASCADE
    ON UPDATE CASCADE;
```

> ⚠️ **CASCADE는 편하지만 위험!** 부모 삭제하면 자식 데이터도 다 날아감. 실무에서는 신중하게 사용

---

## 4. CHECK 제약 조건

> **컬럼에 들어갈 수 있는 값의 조건을 지정**

```sql
CREATE TABLE 직원 (
    직원ID INT PRIMARY KEY,
    이름 VARCHAR(50) NOT NULL,
    나이 INT CHECK (나이 >= 18 AND 나이 <= 100),
    급여 INT CHECK (급여 >= 0)
);
```

- 나이에 -5를 넣으려 하면 → **에러!**
- 급여에 음수를 넣으려 하면 → **에러!**
- 애초에 잘못된 데이터가 들어오지 못하게 막는 것

---

## 📌 무결성 종류 정리

| 무결성 종류 | 의미 | 보장하는 제약 조건 |
|-------------|------|-------------------|
| 개체 무결성 | 기본 키는 NULL이나 중복 불가 | PRIMARY KEY |
| 참조 무결성 | 외래 키는 참조하는 테이블에 존재해야 함 | FOREIGN KEY |
| 도메인 무결성 | 컬럼의 값이 정해진 범위 내여야 함 | CHECK, NOT NULL, DEFAULT |
