# 📝 Day 7 (04/05) — 인덱스 심화 (복합 인덱스 + 설계 가이드라인)


## 🔥 오늘 배운 핵심 한 줄 요약

> **인덱스는 "어떤 컬럼에, 어떤 순서로 거는가"가 성능을 결정한다**

---

## 1. 옵티마이저와 인덱스 선택

- **옵티마이저**: SQL 실행 계획을 자동으로 결정하는 엔진
- 인덱스가 여러 개 있으면 옵티마이저가 **가장 효율적인 인덱스를 선택**함
- 항상 최선의 선택을 하지는 않음 → EXPLAIN으로 확인 필요

```sql
EXPLAIN SELECT * FROM 직원 WHERE 이름 = '김민수';
```

---

## 2. 커버링 인덱스

> **쿼리에 필요한 모든 컬럼이 인덱스에 포함**되어 있어 테이블을 안 봐도 되는 경우

```sql
-- 인덱스: (이름, 부서)
SELECT 이름, 부서 FROM 직원 WHERE 이름 = '김민수';
```

- 테이블까지 안 가고 **인덱스만으로 결과 반환** → 매우 빠름
- EXPLAIN에서 **Extra: Using index**로 확인 가능

---

## 3. 복합 인덱스 ⭐ 핵심!

> **두 개 이상의 컬럼을 묶어서 하나의 인덱스로 만드는 것**

```sql
CREATE INDEX idx_부서_이름 ON 직원(부서, 이름);
```

### 복합 인덱스의 핵심 원리 — "왼쪽부터 순서대로"

인덱스가 `(부서, 이름)` 순서일 때:

| 쿼리 | 인덱스 사용 여부 |
|------|-----------------|
| `WHERE 부서 = '개발'` | ✅ 사용 (첫 번째 컬럼) |
| `WHERE 부서 = '개발' AND 이름 = '김민수'` | ✅ 사용 (둘 다 사용) |
| `WHERE 이름 = '김민수'` | ❌ 사용 불가! |
| `WHERE 이름 = '김민수' AND 부서 = '개발'` | ✅ 사용 (옵티마이저가 순서 조정) |

> ⚠️ **복합 인덱스는 왼쪽 컬럼부터 순서대로 사용됨!**
> 두 번째 컬럼만 단독으로 검색하면 인덱스를 탈 수 없음

### 복합 인덱스 순서 정하는 기준

1. **카디널리티가 높은 컬럼**을 앞에 (고유한 값이 많은 것)
2. **WHERE 절에서 자주 사용되는 컬럼**을 앞에
3. **동등 비교(=) 컬럼**을 범위 비교(>, <) 컬럼보다 앞에

```sql
-- 좋은 예: 동등 비교가 앞, 범위가 뒤
CREATE INDEX idx ON 주문(상태, 주문일자);
SELECT * FROM 주문 WHERE 상태 = '완료' AND 주문일자 >= '2026-01-01';
```

---

## 4. 인덱스 설계 가이드라인 ⭐ 면접 대비!

### 인덱스를 걸어야 하는 경우
- **WHERE 절에서 자주 사용되는 컬럼**
- **JOIN의 ON 조건에 사용되는 컬럼**
- **ORDER BY에 사용되는 컬럼**
- **카디널리티가 높은 컬럼** (성별처럼 2종류밖에 없는 건 효과 낮음)

### 인덱스를 걸지 말아야 하는 경우
- **데이터가 적은 테이블** (풀 스캔이 오히려 빠름)
- **INSERT/UPDATE/DELETE가 빈번한 테이블** (인덱스 유지 비용)
- **카디널리티가 낮은 컬럼** (성별, 활성화 여부 등)

---

## 5. 인덱스의 단점과 주의사항

| 단점 | 설명 |
|------|------|
| 저장 공간 | 인덱스도 디스크 공간을 차지함 |
| 쓰기 성능 저하 | INSERT/UPDATE/DELETE 시 인덱스도 함께 갱신 |
| 관리 비용 | 인덱스가 너무 많으면 오히려 성능 저하 |

> 💡 **인덱스는 "읽기는 빨라지고, 쓰기는 느려진다"는 트레이드오프**

### 인덱스가 안 타는 경우 주의!

```sql
-- ❌ 컬럼에 함수/연산 사용
WHERE YEAR(가입일) = 2026    -- 인덱스 안 탐
-- ✅ 범위로 변환
WHERE 가입일 >= '2026-01-01' AND 가입일 < '2027-01-01'

-- ❌ 타입 불일치
WHERE 전화번호 = 01012345678  -- 문자열 컬럼에 숫자로 비교
-- ✅ 타입 일치
WHERE 전화번호 = '01012345678'
```
