## 왜 인덱스 선택이 필요한가?

테이블에 인덱스가 여러 개 있을 때, MySQL은 **"어떤 인덱스를 써야 가장 빠를까"** 를 판단해야함

잘못 고르면 인덱스가 있어도 **풀 테이블 스캔보다 느려질 수 있음**

---

## MySQL이 인덱스를 선택하는 핵심 원리: 비용 기반 최적화 (Cost-Based Optimizer)

MySQL 옵티마이저는 **"어떤 방법이 더 저렴한가(cost)"** 를 수치로 계산해서 비교함

```
실행 방법 A의 cost vs 실행 방법 B의 cost
→ 더 낮은 쪽 선택
```

cost는 크게 두 가지

```
총 비용 = I/O 비용 + CPU 비용
         (디스크 읽기)   (행 비교, 정렬 등)
```

---

## 인덱스 선택의 핵심 지표: 카디널리티 (Cardinality)

### 카디널리티란?

> 해당 컬럼의 **고유한 값의 수**

```sql
-- 예시 테이블: 100만 행
gender    -- 값: M, F  → 카디널리티 2
city      -- 값: 서울, 부산, ...  → 카디널리티 수백
user_id   -- 값: 모두 고유  → 카디널리티 100만
```

### 왜 카디널리티가 중요한가?

인덱스의 목적은 **"범위를 빠르게 좁히는 것"** 

```
카디널리티 높음 (user_id)
→ 인덱스 타면 바로 1~2개 행으로 좁혀짐 → 인덱스 효과 극대화

카디널리티 낮음 (gender)
→ 인덱스 타도 50만 행 → 차라리 풀스캔이 나을 수 있음
```

MySQL은 `SHOW INDEX FROM 테이블명`으로 카디널리티를 확인할 수 있음

```sql
SHOW INDEX FROM users;
-- Cardinality 컬럼 확인
```

---

## 실제 선택 과정: 셀렉티비티 (Selectivity)

카디널리티를 전체 행 수로 나눈 값

```
셀렉티비티 = 카디널리티 / 전체 행 수

user_id: 1,000,000 / 1,000,000 = 1.0   → 완벽한 인덱스
city:    500 / 1,000,000       = 0.0005 → 나쁘지 않음
gender:  2 / 1,000,000         = 0.000002 → 인덱스 거의 의미 없음
```

셀렉티비티가 높을수록 MySQL이 그 인덱스를 선택할 가능성이 높음

## 인덱스가 있어도 안 쓰는 경우 (중요!)


**1. 범위가 너무 넓을 때**
```sql
-- age 인덱스가 있어도
SELECT * FROM users WHERE age > 1;
-- 거의 모든 행이 해당 → 풀스캔이 더 빠름
```

**2. 함수/연산으로 컬럼을 가공할 때**
```sql
-- 인덱스 못 탐
SELECT * FROM users WHERE YEAR(created_at) = 2024;

-- 인덱스 탐
SELECT * FROM users WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31';
```
이유: 인덱스는 **원본 값 기준**으로 정렬되어 있어요. 함수를 씌우면 원본이 아니라 변환된 값이므로 인덱스 구조를 활용할 수 없음

**3. 복합 인덱스에서 순서를 어길 때**
```sql
-- 인덱스: (last_name, first_name)
WHERE first_name = 'John'              -- 인덱스 못 탐 (첫 번째 컬럼 건너뜀)
WHERE last_name = 'Kim'                -- 인덱스 탐
WHERE last_name = 'Kim' AND first_name = 'John'  -- 인덱스 탐
```
이유: B-Tree 인덱스는 **왼쪽부터 순서대로** 정렬되어 있음

## 통계 정보 (Statistics)

카디널리티는 실제 데이터를 **샘플링해서 추정**한 값이에요. 정확한 전수 조사가 아님

```sql
-- 통계 정보 강제 갱신
ANALYZE TABLE users;
```

데이터가 대량으로 변경된 후 옵티마이저가 이상한 선택을 한다면, 통계 정보가 오래된 것일 수 있음

```
카디널리티 기반으로 추정해서, 비용이 가장 낮은 인덱스를 선택한다.
```
