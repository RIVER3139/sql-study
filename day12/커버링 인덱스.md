# 커버링 인덱스 (Covering Index)

## 핵심 한 줄 요약

> **인덱스만으로 쿼리의 답을 낼 수 있는 인덱스**
> → 테이블에 접근하지 않아도 된다

---

## 일반 인덱스의 동작 방식

```
인덱스 (name 컬럼 기준)
┌──────────┬────────────────┐
│  name    │  row 포인터    │
├──────────┼────────────────┤
│ Alice    │  → 3번 행      │
│ Bob      │  → 1번 행      │
│ Charlie  │  → 5번 행      │
└──────────┴────────────────┘
```

아래 쿼리를 실행하면:

```sql
SELECT age FROM users WHERE name = 'Bob';
```

1. `name` 인덱스에서 'Bob' 찾음 → "1번 행에 있구나"
2. **실제 테이블로 이동** (랜덤 I/O 발생)
3. 1번 행에서 `age` 값 읽어옴

→ **인덱스 → 테이블, 두 번 접근**

---

## 커버링 인덱스의 동작 방식

인덱스를 `(name, age)` 복합 인덱스로 만들면:

```
인덱스 (name, age)
┌──────────┬─────┬────────────────┐
│  name    │ age │  row 포인터    │
├──────────┼─────┼────────────────┤
│ Alice    │  30 │  → 3번 행      │
│ Bob      │  25 │  → 1번 행      │
│ Charlie  │  40 │  → 5번 행      │
└──────────┴─────┴────────────────┘
```

같은 쿼리를 실행하면:

1. 인덱스에서 'Bob' 찾음
2. `age`도 인덱스 안에 있음
3. **테이블 접근 불필요** ✅

---

## 왜 빠른가? (원리)

테이블 접근이 느린 이유는 **랜덤 I/O** 때문이다.

| | 접근 방식 | 속도 |
|---|---|---|
| 인덱스 | 정렬되어 있어 순차 읽기 | 빠름 ✅ |
| 테이블 | 데이터가 흩어져 있어 랜덤 읽기 | 느림 ❌ |

**책 색인으로 비유하면:**
- 일반 인덱스: 색인에서 페이지 번호 확인 → 해당 페이지로 이동해서 내용 확인
- 커버링 인덱스: 색인 자체에 필요한 내용이 다 있음 → 페이지 이동 불필요

---

## 주의: 커버링 인덱스는 "구조"가 아니라 "상황"

같은 인덱스라도 쿼리에 따라 커버링이 되기도, 안 되기도 한다.

```sql
-- 인덱스: (name, age, email)

-- ✅ 커버링 인덱스 (name, age 모두 인덱스에 있음)
SELECT age FROM users WHERE name = 'Bob';

-- ❌ 커버링 인덱스 아님 (phone이 인덱스에 없음)
SELECT phone FROM users WHERE name = 'Bob';
```

**"쿼리에 필요한 모든 컬럼이 인덱스 안에 있을 때"** 커버링 인덱스라고 부른다.

---

## 정리

- 커버링 인덱스 = 인덱스만으로 쿼리의 답이 나오는 상태
- 핵심 이점 = 테이블 랜덤 I/O 제거
- 멀티컬럼 인덱스를 활용하지만, 멀티컬럼 인덱스 = 커버링 인덱스는 아님
